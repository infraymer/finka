# PRD: SaaS-сервис управленческого финансового учета (РФ)

## 1. Обзор продукта

**Рабочее название:** Finka (условно)  
**Тип продукта:** SaaS  
**Регион:** Россия  
**Платформа:** Web-приложение с мобильной адаптацией  
**Технологический стек:** React + Bun + Convex (полная детализация в разделе 11)

### 1.1 Проблема
Собственники и ответственные сотрудники малого и среднего бизнеса не имеют единой и простой системы для ежедневного контроля движения денег, предотвращения кассовых разрывов и регулярного анализа ДДС и P&L.

### 1.2 Цель MVP
Дать пользователям понятный инструмент для:
- фиксации фактических денежных операций;
- планирования будущих поступлений/списаний;
- контроля кассовых разрывов через платежный календарь;
- получения базовой управленческой отчетности (ДДС, P&L, План/Факт).

---

## 2. Целевая аудитория и роли

### 2.1 Сегмент
- Собственники бизнеса
- Ответственные сотрудники за финансы

### 2.2 Роли в MVP
- **Администратор** (единственная роль в MVP):
  - управляет данными компании;
  - создает и редактирует транзакции;
  - ведет справочники;
  - просматривает отчеты.

---

## 3. Scope MVP

### 3.1 Входит в MVP
1. Мульти-тенантность: работа с несколькими компаниями в одном аккаунте.
2. Учет транзакций:
   - поступления;
   - списания;
   - переводы между счетами.
3. Поддержка:
   - фактических операций;
   - плановых операций;
   - регулярных операций.
4. Справочники:
   - проекты (у транзакции только один проект);
   - контрагенты (клиент, сотрудник, подрядчик/поставщик);
   - статьи ДДС.
5. Учет денежных сущностей через универсальную сущность «Счет»
   (банковский счет/наличные/касса).
6. Отчеты:
   - ДДС;
   - P&L;
   - План/Факт;
   - платежный календарь.
7. Ручной ввод операций через UI.
8. Валюта: только RUB.

### 3.2 Не входит в MVP
- Интеграции с банками, 1С, ERP, CRM.
- API для внешних систем.
- Хранение вложений по транзакциям.
- Учет договоров/счетов первички.
- Согласование операций (workflow).
- Журнал аудита изменений.
- Бюджетные лимиты и алерты.
- Налоговый учет и НДС-аналитика.
- Сложная бухгалтерская (double-entry) модель.

## 4. Функциональные требования

## 4.1 Транзакции (ключевая сущность)

### 4.1.1 Типы транзакций
- **Поступление**
- **Списание**
- **Перевод** (между двумя счетами)

### 4.1.2 Обязательные поля транзакции
- ID
- Компания (tenant)
- Тип транзакции
- Статус: план / факт
- Дата операции (для факта) или плановая дата (для плана)
- Сумма (в RUB)
- Счет-источник (для списания и перевода)
- Счет-получатель (для поступления и перевода)
- Статья ДДС
- Проект (опционально, не более одного)
- Контрагент (опционально)
- Комментарий (опционально)
- Признак регулярности (да/нет)
- Параметры регулярности (если да): период, дата начала, дата окончания/без окончания

### 4.1.3 Бизнес-правила
- Каждая транзакция принадлежит ровно одной компании.
- Все суммы хранятся в рублях.
- Для перевода обязательны два разных счета.
- Одна транзакция может быть связана максимум с одним проектом.
- Статья ДДС обязательна для поступлений и списаний.
- Плановые и регулярные операции попадают в платежный календарь.

## 4.2 Справочники

### 4.2.1 Проекты
- Создание/редактирование/архивация проекта.
- Статус проекта: активный/архивный.
- Привязка транзакций к проекту.

### 4.2.2 Контрагенты
- Типы: клиент, сотрудник, подрядчик/поставщик.
- Создание/редактирование/архивация.
- Привязка к транзакциям.

### 4.2.3 Статьи ДДС
- Иерархия (опционально v1.1, плоский список в MVP).
- Тип статьи: доход/расход.
- Использование в аналитике ДДС и План/Факт.

### 4.2.4 Счета
- Универсальная сущность счета (банк, наличные, касса).
- Создание/редактирование/архивация.
- Отображение текущего остатка на основе фактических операций.

## 4.3 Планирование и регулярные операции
- Пользователь может создавать плановые операции на будущие даты.
- Пользователь может создавать регулярные шаблоны:
  - ежемесячно (обязательно для MVP);
  - еженедельно/ежеквартально (опционально, если не усложняет реализацию).
- Система генерирует будущие экземпляры операций в рамках горизонта платежного календаря.

## 4.4 Отчеты

### 4.4.1 ДДС
- Факт за выбранный месяц.
- План за выбранный месяц.
- Разрезы: по статьям ДДС, проектам, счетам.

### 4.4.2 P&L (упрощенный управленческий)
- Доходы и расходы по выбранному месяцу.
- Основан на транзакциях и статьях.
- Без налоговых корректировок и сложной бухгалтерской логики.

### 4.4.3 План/Факт
- Сравнение плановых и фактических значений за месяц.
- Базовые отклонения: абсолютное и процентное.

### 4.4.4 Платежный календарь
- Месячный вид.
- Показывает плановые и регулярные будущие операции.
- Подсвечивает дни/периоды с потенциальным отрицательным остатком.

---

## 5. Пользовательские сценарии (MVP)

1. Администратор создает компанию, счета, статьи ДДС, контрагентов, проекты.
2. Администратор заносит фактические поступления/списания и переводы.
3. Администратор заводит плановые платежи на месяц вперед.
4. Администратор настраивает регулярные платежи (например, аренда, зарплата).
5. Администратор открывает платежный календарь и видит риск кассового разрыва.
6. Администратор анализирует месяц в ДДС, P&L и План/Факт.

---

## 6. Нефункциональные требования

- SaaS-архитектура с изоляцией данных по tenant (компаниям).
- Адаптивный интерфейс для desktop/mobile web.
- Базовая производительность для интерактивной работы с отчетами за месяц.
- Надежное хранение транзакций и идемпотентность сохранения при повторной отправке формы.
- Базовая безопасность: аутентификация и авторизация в пределах роли админа.

---

## 7. Ограничения и допущения

- В MVP нет интеграций: все данные вводятся вручную.
- Точность P&L ограничена упрощенной моделью и может отличаться от бухгалтерской отчетности.
- Продукт ориентирован на управленческий учет, а не на регламентированный бухгалтерский.
- Масштаб нагрузки будет уточнен после пилотных запусков.

---

## 8. Метрики успеха MVP (предложение для согласования)

> Заказчик пока не зафиксировал KPI. Ниже стартовые метрики для валидации MVP.

- Доля активных компаний, которые еженедельно вносят операции.
- Количество компаний, использующих платежный календарь не реже 2 раз в неделю.
- Время подготовки ежемесячного управленческого отчета (до/после внедрения).
- Процент плановых операций, переведенных в факт в течение месяца.

## 11. Полный технологический стек

### 11.1 Frontend
- Язык: **TypeScript** (`strict: true`).
- UI: **React 18**.
- Runtime и пакетный менеджер: **Bun**.
- Роутинг: **React Router**.
- Работа с данными Convex: **`convex/react`** (`useQuery`, `useMutation`).
- Формы и валидация: **React Hook Form + Zod**.
- UI-компоненты: **shadcn/ui** (на базе Radix UI).
- Стилизация: **Tailwind CSS**.
- Таблицы: **TanStack Table**.
- Графики: **Recharts**.
- Даты/таймзоны: **date-fns + date-fns-tz** (`Europe/Moscow`).
- Форматирование валюты: **Intl.NumberFormat** (`ru-RU`, `RUB`).

### 11.2 Backend / BaaS
- Платформа: **Convex**.
- Модель данных: **Convex Schema** (`convex/schema.ts`).
- Бизнес-логика и API: **queries/mutations/actions** Convex.
- Валидация: **Convex validators** + Zod (в сложных DTO).
- Индексы под основные фильтры: `organizationId`, `date`, `type`, `projectId`, `counterpartyId`, `cashflowCategoryId`.
- Multi-tenant изоляция: любой запрос обязан фильтровать по `organizationId`.

### 11.3 Auth и доступ
- Auth в MVP: **Convex Auth** (email/password или magic link).
- Роли в MVP: только **admin**.
- Пользователь может иметь доступ к нескольким организациям, активная организация выбирается в UI.

### 11.4 Финансовые и временные стандарты
- Валюта MVP: только **RUB**.
- Хранение сумм: `amountKopecks` (**integer**, без float).
- Отображение сумм: в рублях с 2 знаками после запятой.
- Таймзона продукта: **Europe/Moscow**.
- Горизонт и шаг базовой отчетности: календарный месяц.

### 11.5 Качество и процессы разработки
- Линтинг: **ESLint**.
- Форматирование: **Prettier**.
- Проверка типов: `tsc --noEmit`.
- Тесты: **Vitest + React Testing Library** (frontend) и unit-тесты бизнес-правил на уровне Convex-функций.
- Git hooks (рекомендовано): **Husky + lint-staged**.
- CI (рекомендовано): **GitHub Actions** (lint + typecheck + tests).

### 11.6 Эксплуатация и наблюдаемость
- Логирование: structured logs в Convex-функциях.
- Мониторинг ошибок (рекомендовано): **Sentry** (frontend + backend functions).
- Продуктовая аналитика (опционально после стабилизации MVP): **PostHog** или **Яндекс Метрика**.

### 11.7 Окружения и деплой
- Окружения: `local` / `staging` / `production`.
- Конфигурация: `.env` + секреты окружений в CI/Convex.
- Frontend deploy: **Vercel** / **Netlify** (или аналогичный managed hosting).
- Backend deploy: **Convex Cloud**.

## 12. Подход к разработке на Convex.dev

### 12.1 Принципы
- Backend-first через Convex: сначала schema и инварианты, затем UI.
- Источник истины для бизнес-правил — Convex mutations/queries.
- Реактивность данных используется как основной механизм обновления UI.

### 12.2 Engineering workflow
1. Спроектировать коллекции и поля в `convex/schema.ts`.
2. Заложить индексы под реальные фильтры экранов.
3. Реализовать mutation/query с tenant-проверкой (`organizationId`).
4. Подключить UI через `useQuery`/`useMutation`.
5. Покрыть критичные инварианты unit-тестами.

### 12.3 Обязательные требования к реализации
- Все денежные суммы — `amountKopecks` (integer).
- Серверная валидация обязательна для всех mutation.
- `transfer` всегда между двумя разными счетами.
- Любой доступ к данным ограничен активной организацией пользователя.

### 12.4 Критерии архитектурной готовности фичи
- Есть индексы для всех пользовательских фильтров.
- Валидации и ошибки описаны на сервере и обработаны в UI.
- UI не дублирует бизнес-логику, только отображает/редактирует данные.
- Тесты проверяют ключевые финансовые инварианты.
