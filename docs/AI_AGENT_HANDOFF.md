## 2) Утвердить доменную модель в виде «контракта»

ИИ-агенту нужен неизменный контракт сущностей, чтобы не переделывать миграции и UI.

### Минимальные сущности MVP
1. `organizations` (тенант)
2. `users` (пока роль `admin`)
3. `accounts` (счет/касса — единая сущность)
4. `projects`
5. `counterparties` (`client | employee | contractor`)
6. `cashflow_categories` (статьи ДДС)
7. `transactions` (`income | expense | transfer`)
8. `recurring_rules` (регулярные операции)
9. `planned_transactions` (плановые операции)

### Обязательные поля `transactions`
- `organizationId`
- `type` (`income | expense | transfer`)
- `amountKopecks` (> 0)
- `date` (дата операции)
- `accountIdFrom` (для `expense`/`transfer`)
- `accountIdTo` (для `income`/`transfer`)
- `projectId` (опционально)
- `counterpartyId` (опционально)
- `cashflowCategoryId` (обязательно для `income`/`expense`, опционально для `transfer`)
- `isPlanned` (bool)
- `status` (`planned | posted`)
- `comment` (опционально)
- `createdAt`, `updatedAt`

## 3) Прописать жесткие бизнес-правила (DoD для агента)

1. `transfer` не влияет на итоговые притоки/оттоки в ДДС (только движение между счетами).
2. Для `income` обязательно `accountIdTo`.
3. Для `expense` обязательно `accountIdFrom`.
4. Для `transfer` обязательны оба счета и они должны быть разными.
5. Все выборки строго в рамках `organizationId` (multi-tenant isolation).
6. Удаление справочника запрещено, если он используется в транзакциях (или soft delete).
7. Любая сумма хранится только в копейках (integer).

## 4) Сформировать приоритетный backlog для первой итерации

Чтобы агент писал код поэтапно, нужен список задач с четким порядком.

### Sprint 1 (Foundation)
- Инициализация проекта (Bun + React + Convex + TypeScript + lint/format).
- Базовая авторизация и привязка пользователя к `organization`.
- Convex schema + индексы для всех MVP-сущностей.
- Seed-данные для локальной разработки.

### Sprint 2 (Core CRUD)
- CRUD: `accounts`, `projects`, `counterparties`, `cashflow_categories`.
- Список транзакций + фильтры (период, тип, проект, контрагент, статья).
- Создание транзакции (`income/expense/transfer`) с серверной валидацией.

### Sprint 3 (Planning)
- Плановые операции.
- Регулярные операции (генерация будущих плановых транзакций).
- Платежный календарь (месяц).

### Sprint 4 (Reports MVP)
- Отчет ДДС (месяц, группировка по статьям).
- P&L (упрощенный, на базе категорий/правил маппинга).
- План/Факт (по месяцу).

- [ ] Учтен `organizationId` во всех запросах.

## 7) Что дать агенту прямо сейчас (чеклист входа в разработку)

1. Подтвержденный список сущностей и полей (из раздела 2).
2. Решения по UI/библиотекам (раздел 1).
3. Первый тикет: **"Scaffold проекта + Convex schema + индексы + seed"**.
4. Ограничение на размер PR: до 400–600 строк или 1 функциональный вертикальный срез.
5. Явное требование: в конце каждого PR агент пишет:
   - что сделано;
   - какие команды запускал;
   - что не покрыто и почему.

## 9) Подход к разработке на Convex.dev (как работать правильно)

Ниже практический подход, который стоит считать стандартом команды для задач в этом проекте.

### 9.1 Convex-first архитектура
- Сначала проектируются **данные и инварианты** в `convex/schema.ts`.
- Затем пишутся **mutation/query** как единственный источник бизнес-логики.
- UI только вызывает Convex-функции и не хранит критичную бизнес-логику локально.

### 9.2 Логика по слоям
1. `schema.ts` — модель и обязательные поля.
2. `mutations` — изменение состояния + проверки инвариантов.
3. `queries` — чтение и агрегации под конкретные экраны.
4. `actions` — только для внешних интеграций/долгих операций (не для обычного CRUD).

### 9.3 Правила работы с данными
- Любая mutation повторно валидирует входные данные (даже если UI уже проверил форму).
- Любой query/mutation принимает `organizationId` и проверяет доступ текущего пользователя к организации.
- Денежные поля — только integer в копейках (`amountKopecks`).
- Для фильтруемых списков сначала проектируется индекс, потом запрос.

### 9.4 Реактивный UI-подход
- Экран списка строится на `useQuery` (данные реактивно обновляются сами).
- Формы изменения данных отправляют `useMutation`.
- После мутаций не делаем ручной refetch «как в REST», полагаемся на реактивность Convex.

### 9.5 Транзакционные инварианты (обязательно на сервере)
- `transfer`:
  - оба счета обязательны;
  - `accountIdFrom != accountIdTo`;
  - не влияет на net inflow/outflow ДДС.
- `income` требует `accountIdTo`, `expense` требует `accountIdFrom`.
- Нельзя удалить справочник, если на него есть ссылки в транзакциях (или использовать soft delete).

### 9.6 Порядок реализации фичи на Convex
1. Описать/обновить схему.
2. Добавить индексы под будущие выборки.
3. Реализовать мутации с полной валидацией.
4. Реализовать запросы под экран.
5. Подключить UI к `useQuery/useMutation`.
6. Добавить тесты бизнес-правил.
7. Прогнать lint/typecheck/tests.

### 9.7 Антипаттерны (чего избегать)
- Проверки только в UI без серверной валидации.
- Сложные вычисления отчетов прямо в React-компонентах вместо Convex-query.
- Отсутствие индекса при фильтрации по большим коллекциям.
- Использование float для денег.
- Игнорирование tenant-изоляции хотя бы в одном query.
